## Vault Providers {{ id: 'vault-providers' }}

Vault provider nodes configure where your bot resolves secrets. Drag a vault provider onto the canvas, configure it, and connect it to any node that needs secrets (TOTP Generate, Get Secret) via the purple **Vault** handle.

Each vault provider has a purple **To Vault** handle at the top. Consumer nodes (TOTP Generate, Get Secret) have a purple **Vault** handle at the bottom. Connect them to establish the vault relationship.

```
┌──────────────────┐
│  Azure Key Vault │
│  secrets.azure   │
│   ◆ To Vault     │  ← purple handle (top)
└───────┬──────────┘
        │ (purple connection)
┌───────┴──────────┐
│  TOTP Generate   │
│  security.totp   │
│   ◆ Vault        │  ← purple handle (bottom)
└──────────────────┘
```

**Available Vault Providers:**

| Provider | Best for | Authentication |
|----------|----------|---------------|
| Orchestrator Vault | Production bots managed by Skuld | Runner token or API key |
| Azure Key Vault | Azure cloud deployments | Managed Identity or Service Principal |
| AWS Secrets Manager | AWS cloud deployments | IAM Role or Access Keys |
| GCP Secret Manager | Google Cloud deployments | Application Default Credentials or Service Account |
| HashiCorp Vault | On-premise enterprise | Token or AppRole |
| Local Vault | Development and standalone runners | Master password (AES-256-GCM) |

<NodeHeader id="secrets-orchestrator_vault" icon="KeyRound" label="Orchestrator Vault" description="Connect to the Skuld Orchestrator vault for production secrets" bgColor="#fee2e2" textColor="#991b1b" />

<Properties>
  <Property name="type" type="string">
    `secrets.orchestrator_vault`
  </Property>
</Properties>

The default vault for production bots. Secrets are managed centrally in the Skuld Orchestrator and resolved at runtime by the BotRunner.

**Configuration:**
- **orchestrator_url** (`text`, required) — URL of the Skuld Orchestrator. Can also be set via `SKULDBOT_ORCHESTRATOR_URL` environment variable.
- **auth_method** (`select`, default: `runner_token`) — How the runner authenticates:
  - `runner_token` — Token from `SKULDBOT_ORCHESTRATOR_TOKEN` env var
  - `api_key` — API key authentication
- **secrets** (`textarea`) — Secret names to load, one per line

**Output:**
- **loaded** — Number of secrets loaded
- **secretNames** — Names of loaded secrets

<Properties>
  <Property name="type" type="string">

Unlock the local encrypted vault on the BotRunner. Uses AES-256-GCM encryption with PBKDF2-SHA256 key derivation (600,000 iterations). Ideal for development, standalone bots, and on-premise deployments without external vault infrastructure.

**Configuration:**
- **vault_path** (`text`, default: `.skuldbot`) — Path to vault directory
- **secrets** (`textarea`) — Specific secrets to load (optional, loads all if empty)

**Output:**
- **unlocked** — Vault unlock status
- **secretCount** — Number of secrets loaded
- **secretNames** — Names of loaded secrets

<Note>
  The Local Vault master password is provided via the `SKULDBOT_VAULT_PASSWORD` environment variable. Never hardcode it in the workflow.
</Note>

<NodeHeader id="security-totp_generate" icon="ShieldCheck" label="TOTP Generate" description="Generate MFA code for bot authentication" bgColor="#fee2e2" textColor="#991b1b" />

<Properties>
  <Property name="type" type="string">
    `security.totp_generate`
  </Property>
</Properties>

Generates a TOTP (Time-based One-Time Password) code for MFA authentication. The bot generates the same 6-digit code that Google Authenticator, Microsoft Authenticator, or any TOTP-compatible app produces.

Connect a vault provider node via the purple **Vault** handle to provide the shared secret.

**Configuration:**
- **secret_name** (`text`, required) — Name of the TOTP shared secret in the connected vault (e.g., `erp-sap-mfa-secret`)
- **mfa_type** (`select`, default: `google_authenticator`) — Authenticator type:
  - `google_authenticator` — Google Authenticator
  - `microsoft_authenticator` — Microsoft Authenticator
  - `authy` — Authy
  - `duo` — Duo Security
  - `okta_verify` — Okta Verify
  - `custom` — Custom TOTP (shows advanced options: digits, period, algorithm)

**Output:**
- **code** (`string`) — The generated TOTP code (e.g., `847293`)
- **expiresIn** (`number`) — Seconds until this code expires

**Example — Login to SAP with MFA:**

```
[Manual Trigger]
    |
[Local Vault] ─── vault connection ───┐
    |                                  |
[Open Browser] -> https://sap.com     |
    |                                  |
[Fill Field] -> #username              |
    |                                  |
[Fill Field] -> #password              |
    |                                  |
[Click] -> #login-button               |
    |                                  |
[Wait Element] -> #mfa-code-input      |
    |                                  |
[TOTP Generate] <- vault connection ──┘
    |              secret: erp-sap-mfa-secret
    |              type: Google Authenticator
    |
[Fill Field] -> #mfa-code -> ${TOTP Generate.code}
    |
[Click] -> #verify-button
    |
[Wait Element] -> #dashboard  ← logged in!
```

**Setup (one-time):**
1. Create a service account on the target system with TOTP MFA enabled
2. When the system shows the QR code, extract the shared secret
3. Store it in your vault under a name like `erp-sap-mfa-secret`
4. Multiple bots can use the same secret — just reference the same name

<Note>
  The TOTP code changes every 30 seconds. The bot generates it at the exact moment it fills the MFA field. Works with any system that supports TOTP (RFC 6238) — Microsoft 365, Google Workspace, Salesforce, SAP, AWS, Okta, GitHub, and hundreds more.
</Note>

**Security:**
- The shared secret is encrypted at rest in the vault (AES-256-GCM)
- It lives in memory only during code generation and is scrubbed immediately after
- The code is redacted in logs and Evidence Pack as `[TOTP_CODE_REDACTED]`
- Access to the secret is controlled by vault policies per bot
- Every access generates an audit event

**Supported Authenticators:**

| Authenticator | Supported | Standard |
|--------------|:---------:|----------|
| Google Authenticator | Yes | TOTP RFC 6238 |
| Microsoft Authenticator | Yes | TOTP RFC 6238 |
| Authy | Yes | TOTP RFC 6238 |
| Duo Security | Yes | TOTP mode |
| Okta Verify | Yes | TOTP RFC 6238 |
| Any TOTP app | Yes | RFC 6238 |
| Push notifications | No | Requires human interaction |
| Hardware tokens (YubiKey) | No | Requires physical device |

## Best Practices

<Warning>
  **Never hardcode credentials** in your bot configurations. Always use a vault provider and reference secrets by name.
</Warning>

### Vault Selection Guide

| Scenario | Recommended Vault |
|----------|------------------|
| Local development / debugging | Local Vault |
| Production with Skuld Orchestrator | Orchestrator Vault |
| Azure cloud deployment | Azure Key Vault |
| AWS cloud deployment | AWS Secrets Manager |
| Google Cloud deployment | GCP Secret Manager |
| On-premise with existing HashiCorp | HashiCorp Vault |
| On-premise without vault infra | Local Vault or Orchestrator (embedded) |

### Recommended Secret Organization

```
vault/
├── mfa/
│   ├── erp-sap-mfa-secret
│   ├── crm-salesforce-mfa-secret
│   └── portal-banco-mfa-secret
├── api-keys/
│   ├── openai-api-key
│   ├── stripe-api-key
│   └── sendgrid-key
├── database/
│   ├── prod-db-password
│   └── staging-db-password
├── oauth/
│   ├── google-client-secret
│   └── ms365-client-secret
└── services/
    ├── twilio-auth-token
    └── aws-secret-key
```
